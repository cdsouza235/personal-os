## The contract (D-PO-010)

```
compute_due_and_owed(routines, completions, *, as_of_date, occurrence_overrides={})
```

- `routines`: a list of routine dicts, the same shape `list_routines`/`get_routine` return
  (`routine_id`, `name`, `status`, `enabled`, `cadence_type`, `cadence_config`, `rotation_group`,
  `weekly_target`, ...).
- `completions`: a list of completion dicts, the same shape `list_routine_completions` returns
  (`routine_id`, `completed_for_date`, ...).
- `as_of_date`: an ISO date string (or `datetime.date`, your choice — document it), the date to
  compute due-today for.
- `occurrence_overrides`: optional, keyed by `(routine_id, due_date)` (or an equivalent stable key
  — document your choice), carrying a per-occurrence missed-behavior override that beats
  `missed_behavior_default` for that one occurrence only (this is how the cleaning pool's
  reply-driven missed-handling works — see D-PO-010).
- Returns: the due-today set for `as_of_date`, plus any "owed" make-up debt from weekly-target
  shortfalls (exact return shape is your implementation choice — a dataclass or a dict, either is
  fine — but it must be documented and every field must be exercised by a test).
- Pure function: no I/O, no mutation of its inputs, deterministic — same inputs always produce the
  same output. Only consider `status == "active"` and `enabled == True` routines as due-eligible;
  everything else is never due (paused/archived/disabled routines are simply absent from the
  due-today set, not an error).

## Cadence types — due-today semantics

Fixed ISO week for ALL weekly accounting: Monday–Sunday, resetting at the Monday boundary
regardless of when in the prior week a target was met (D-PO-010). "This week" = the Mon–Sun week
containing `as_of_date`.

- `daily` — due every day.
- `weekdays` — due Monday–Friday only.
- `every_n_days` — due every N days from some anchor (last completion date, or routine creation
  date if never completed — pick one, document it); `cadence_config` carries N (key your own
  choice, e.g. `{"n": 3}` — document it).
- `specific_days` — due on specific weekdays named in `cadence_config` (e.g.
  `{"days": ["mon", "thu"]}` — document your exact key/value convention).
- `manual_only` — never appears in the due-today set (no automatic scheduling at all; it exists
  purely so the routine record can exist without the engine ever surfacing it).
- `x_times_per_week` — due N times per week, order-independent, `weekly_target` carries N. Treat
  this as due on any day of the current week where completions-so-far-this-week < N; once N is met
  for the week, not due again until next Monday. (This baseline PRD §3.1 type and the newer
  `weekly_target_count` below are semantically the same "N times, any day" shape — implement them
  identically, sharing logic; note this explicitly in your handoff rather than silently guessing a
  difference the design doc doesn't actually specify.)
- `weekly_target_count` — same due-today logic as `x_times_per_week` above (N completions anywhere
  in the week, order-independent; `weekly_target` or `cadence_config` carries N — pick one source
  of truth and document it, preferring `weekly_target` since it's a first-class column).
- `weekly_target_reps` — a rep/quantity target per week (not just a completion count) — e.g.
  Grease-the-Groove wants 45 reps/exercise/week. Each completion may itself carry a rep count (in
  the completion's `metadata`, e.g. `{"reps": 15}` — document your exact key). Due-today: still due
  on any day this week where total reps-so-far-this-week < target; "owed" debt for this type is the
  shortfall in REPS, not completion count.
- `rotating_sequence` — a generic ordered pool of routines sharing one `rotation_group`; the pool
  advances one routine per due occurrence (whichever pool member is "next" becomes due, the others
  in the same group are not). Order = a stable, deterministic ordering rule you choose and document
  (e.g. `routine_id` sort order, or explicit sequence position if you add one — do NOT add new
  schema columns for this without flagging it, since state.py should not need changes; a documented
  ordering derived from existing fields is preferred).
- `rotating_weekday_pool` — like `rotating_sequence` but bound to specific weekdays (distinct per
  D-PO-010 from the generic `rotating_sequence`): the pool advances through its members, but only
  on the weekdays the pool is configured for (`cadence_config`, document your key convention).
- Cleaning-pool-shaped usage (15-20 tasks, one `rotation_group`, e.g. `"cleaning"`, cadence type
  `rotating_sequence` or `rotating_weekday_pool` per how it's actually configured) is not a
  separate cadence type — it's a `rotating_sequence`/`rotating_weekday_pool` instance whose missed
  handling is driven dynamically by `occurrence_overrides` rather than `missed_behavior_default`
  (see D-PO-010) — the engine doesn't need to know "this pool is cleaning," it just needs
  `occurrence_overrides` to work correctly for ANY rotation-group routine.

## Missed-behavior types

Applies when a routine WAS due on some past date (before `as_of_date`) and has no completion for
that due date. `missed_behavior_default` on the routine picks the default handling;
`occurrence_overrides[(routine_id, due_date)]` (if present) overrides it for that one occurrence
only.

- `combine_with_next` — the missed occurrence's obligation folds into the next due occurrence (no
  separate "owed" tracking; the next due date just... is still due, nothing extra).
- `bump_schedule_by_one_day` — the routine's next due date shifts forward by one day rather than
  staying on its original cadence.
- `carry_forward_within_week` — the missed occurrence becomes "owed" debt that must be made up
  before the current ISO week ends; if the week ends without makeup, the debt does NOT carry into
  next week (bounded to the week it was missed in).
- `skip_and_continue` — the missed occurrence is simply dropped; no debt, no schedule shift, the
  routine's next occurrence proceeds on its normal cadence as if nothing happened.
- `escalate_to_review` — the missed occurrence generates an "owed"/flagged entry that surfaces for
  human review (Shutdown/Review routine territory) rather than being silently resolved by the
  engine; document what field/shape signals this escalation in your return value.

## Seed data (D-PO-010, for your table-driven tests — these are the REAL production routines)

Cleaning (rotating pool, 1/day weekdays) · Reading (4x/wk, `weekly_target_count`) ·
Prayer/Meditation (2x/wk, `weekly_target_count`) · Grease-the-Groove (per-exercise rows,
`rotation_group="gtg"`, `weekly_target_reps`, 45 reps/exercise/wk) · Shutdown/Review (daily
evening, `daily`). (Fitness/Strength is tracked externally and NOT part of this engine — do not
add fitness-specific logic.)

## Testing (this is the point of the packet — be exhaustive, table-driven)

- Every cadence type needs its own due/not-due test matrix (at minimum: due when eligible, not due
  when already completed today, not due when disabled/paused/archived, not due when
  `manual_only`).
- Weekly-target types (`x_times_per_week`, `weekly_target_count`, `weekly_target_reps`): test the
  Monday-boundary reset explicitly (completions from last week must NOT count toward this week's
  target, even if the trailing 7 days would include them) — this is the one detail most likely to
  be gotten wrong, test it directly with dates that straddle a week boundary.
- Rotation types: test that only the "next" pool member is due, not the whole pool; test advancement
  across multiple due occurrences.
- Every missed-behavior type needs a test proving its specific handling (debt shape, schedule
  shift, or lack thereof).
- `occurrence_overrides` needs a test proving it overrides `missed_behavior_default` for exactly
  the targeted occurrence and does not affect any other occurrence of the same or a different
  routine.
- Use the seed routines above as fixtures for at least one integration-shaped test per routine,
  in addition to the type-by-type unit matrix.

Run the full suite before reporting (`PYTHONPATH=src python3 -m unittest discover -s tests -p
'test_*.py'`) — it must be genuinely green, not just your new tests.

Report `status: pass` with an honest, complete handoff summary listing every documented convention
choice you made (data key names, ordering rules, return-value shape) so the auditor can verify them
against this task rather than having to guess your intent.
